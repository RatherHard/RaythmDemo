# 谱面文件格式定义

本文档定义 RaythmDemo 谱面文件的当前设计草案。格式目标是用 JSON 描述歌曲元数据、BPM 时间线与音符列表，为后续 `Game` 模块的谱面解析、时间换算、判定逻辑和渲染推进提供统一输入。

> 当前文档优先固化最小可玩切片所需字段。未在本文档中定义的字段应视为扩展字段，解析器可以忽略，但不得依赖其存在。
> 谱面文件应使用标准 JSON 与 UTF-8 编码；示例中的注释性说明不属于文件内容。

---

## 1. 顶层结构

谱面文件顶层是一个 JSON 对象，包含以下字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `meta` | object | 是 | 歌曲与谱面元数据 |
| `time` | array | 是 | BPM 变化时间线 |
| `note` | array | 是 | 音符列表 |

最小示例：

```json
{
    "meta": {
        "title": "Example Song",
        "artist": "Example Artist",
        "creator": "Example Charter",
        "path": "songs/example.ogg",
        "offset": 0
    },
    "time": [
        {
            "beat": [0, 0, 1],
            "bpm": 120.0
        }
    ],
    "note": [
        {
            "beat": [4, 0, 4],
            "column": 3
        },
        {
            "beat": [4, 2, 4],
            "endbeat": [5, 0, 4],
            "column": 1
        }
    ]
}
```

---

## 2. 节拍位置 `beat`

所有与谱面时间位置相关的字段都使用 `beat` 表示。`beat` 是长度为 3 的整数数组：

```json
[x, y, z]
```

含义：

| 下标 | 名称 | 说明 |
|------|------|------|
| `x` | 小节编号 | 第 `x` 小节，从 0 开始计数 |
| `y` | 小节内切分位置 | 将当前小节切分为 `z` 等份后，第 `y` 个切分点 |
| `z` | 小节切分数 | 当前小节被切分成的等份数 |

约束：

- `x` 必须为大于或等于 0 的整数。
- `z` 必须为大于 0 的整数。
- `y` 必须满足 `0 <= y < z`。
- 一个小节按基准 BPM 下的四拍计算。
- 同一位置可用不同切分表示，例如 `[1, 2, 4]` 与 `[1, 1, 2]` 表示相同位置；解析器应在内部转换为统一的有理数或 tick 表示后比较。

---

## 3. `meta`

`meta` 存储歌曲和谱面的基础信息。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `title` | string | 是 | 歌曲名称 |
| `artist` | string | 是 | 曲师名称 |
| `creator` | string | 是 | 谱师名称 |
| `path` | string | 是 | 歌曲音频文件路径 |
| `offset` | integer | 是 | 音乐播放向后偏移的时间，单位为毫秒 |

示例：

```json
{
    "meta": {
        "title": "Example Song",
        "artist": "Example Artist",
        "creator": "Example Charter",
        "path": "songs/example.ogg",
        "offset": 0
    }
}
```

说明：

- `path` 是相对资源路径，不应在谱面文件中写入平台相关绝对路径。
- `path` 的实际解析、路径规范化和资源根目录限制由后续资源加载层负责。
- `offset` 用于调整谱面和音乐的对齐。其值为整数，单位为毫秒，表示音乐播放需要相对谱面时间向后偏移的时间。

---

## 4. `time`

`time` 存储歌曲中的 BPM 变化。其值是对象数组，每个对象表示从指定 `beat` 起生效的 BPM。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `beat` | integer[3] | 是 | BPM 变化生效的位置 |
| `bpm` | number | 是 | 从该位置开始应用的 BPM |

示例：

```json
{
    "time": [
        {
            "beat": [0, 0, 1],
            "bpm": 120.0
        },
        {
            "beat": [233, 0, 4],
            "bpm": 150.0
        }
    ]
}
```

约束：

- `time` 至少应包含一个 BPM 段。
- 第一段 BPM 应从 `[0, 0, 1]` 或等价的起始位置生效。
- `bpm` 必须为大于 0 的数值。
- 多个 BPM 段应按 `beat` 升序排列；若输入未排序，解析器应排序后再进行时间换算。
- 不应在同一 `beat` 定义多个不同 `bpm`。若出现冲突，应视为谱面格式错误。

基准 BPM：

- 基准 BPM 用于谱面流速和视觉滚动参考。
- 基准 BPM 定义为：所有 BPM 段中，最接近“按持续时长加权平均 BPM”的一个实际 BPM 值。
- 若无法计算持续时长（例如缺少结束时间），可先使用第一段 BPM 作为临时基准，后续解析器实现时再补齐完整策略。

---

## 5. `note`

`note` 存储谱面中的音符。其值是对象数组。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `beat` | integer[3] | 是 | 音符开始位置 |
| `endbeat` | integer[3] | 否 | Hold 音符结束位置 |
| `column` | integer | 是 | 音符所在轨道 |

示例：

```json
{
    "note": [
        {
            "beat": [4, 0, 4],
            "column": 3
        },
        {
            "beat": [4, 2, 4],
            "endbeat": [5, 0, 4],
            "column": 1
        }
    ]
}
```

音符类型：

- 若对象没有 `endbeat`，该音符为普通 Tap 音符。
- 若对象存在 `endbeat`，该音符为 Hold 音符。
- 当前格式不额外提供 `type` 字段，音符类型仅由是否存在 `endbeat` 区分。

`column` 约束：

- `column` 表示音符所在轨道，从左到右计数。
- 当前 4K 最小可玩切片建议使用 0 基索引：`0`、`1`、`2`、`3` 分别表示从左到右四条轨道。
- 后续若支持其它键型，应由谱面元数据或玩法配置明确轨道数量，再校验 `column` 是否落在有效区间内。

`endbeat` 约束：

- `endbeat` 必须晚于 `beat`。
- Tap 音符不得包含 `endbeat`。
- Hold 音符必须包含 `endbeat`。

---

## 6. 解析与校验建议

解析器读取谱面时应至少执行以下校验：

1. 顶层必须是 JSON 对象。
2. `meta`、`time`、`note` 字段必须存在且类型正确。
3. 所有 `beat` / `endbeat` 必须满足长度与数值范围约束。
4. `time` 必须至少包含一个有效 BPM。
5. BPM 必须大于 0，且同一位置不得定义冲突 BPM。
6. `note` 中所有 `column` 必须落在当前玩法轨道范围内。
7. Hold 音符的 `endbeat` 必须晚于 `beat`。
8. 解析器应将 `beat` 转换为统一内部时间表示，避免直接用浮点数比较节拍位置。

---

## 7. 暂不定义的内容

为避免首个 Game 切片过度设计，以下内容暂不在当前格式中固化：

- 谱面难度、等级、封面图、预览音频等曲库展示字段。
- Offset、变速、停顿、滚速组、镜头、特效等高级时间线。
- 多音轨、音效触发、BGA、皮肤和 UI 配置。
- 编辑器专用字段。
- 完整 JSON Schema 文件。

这些内容应在对应功能进入开发计划后，再以向后兼容的方式扩展。
