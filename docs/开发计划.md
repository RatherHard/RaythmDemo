### 阶段 4：Platform 输入事件与事件泵演进

**状态**：`[x]` 已完成窗口事件泵、设备输入封装与输入状态系统

**目标**

- 在不引入过重消息总线的前提下，建立键盘、鼠标与窗口生命周期事件的统一采集边界。
- 将 SDL 全局事件队列消费职责从 `Window` 中剥离，避免窗口层与输入层争用事件队列。
- 为后续 Core 主循环、输入低延迟处理、输入录制回放和动作映射保留稳定扩展点。

**设计原则**

- 优先新增薄 `Platform::EventPump` 或同等职责类型，而不是新增“总消息处理类”。
- `EventPump` 只负责从 SDL 拉取事件并翻译为项目自有事件，不承载业务决策。
- `Window` 继续作为 SDL 窗口 RAII 封装，长期职责应收敛到窗口资源、窗口 ID、尺寸/显示状态和 Vulkan surface 创建。
- 输入模块应维护 `InputState` 或同等状态快照，而不是只做按键消息回调。
- 当前阶段由 `main` 或未来 `Core::Application` 显式消费事件；仅当多个模块确实需要订阅同一事件流时，再考虑正式 dispatcher/router。

**短期待办**

- [x] 定义轻量窗口事件封装并通过 `Platform::EventPump` 覆盖 close、resize、focus 等窗口生命周期事件。
- [x] 定义轻量输入事件类型，覆盖 keyboard、mouse button、mouse motion。
- [x] 新增 `EventPump` 测试，验证 SDL 窗口事件翻译、未知事件跳过和退出事件处理路径。
- [x] 将 `Window::pollEvent()` 中的 SDL 取事件逻辑迁移到唯一事件泵，并同步更新 `src/main.cpp` 调用方。
- [x] 新增输入状态系统测试，验证 key down/up、mouse button down/up、mouse motion 和帧边沿状态。
- [x] 在最小主循环中显式消费统一平台事件流；窗口事件已转发给渲染器，输入事件已接入 `InputState`，并在 focus lost 时清理 held 输入。

**验证依据**

- `cmake --build /e/RaythmDemo/build --config Debug` 构建通过。
- `ctest --test-dir /e/RaythmDemo/build -C Debug --output-on-failure` 执行通过，3/3 tests passed。
- `ctest --test-dir /e/RaythmDemo/build -C Debug -R Window --output-on-failure` 执行通过，1/1 tests passed。

### 阶段 5：Core 基础设施与应用生命周期收束

**状态**：`[x]` 已完成 Core 主循环、时间系统与应用生命周期基础收束

**目标**

- 建立 `Raythm::Core` 模块基础设施，提供 `Core::Application`、主循环、时间系统和统一应用生命周期管理。
- 将当前 `src/main.cpp` 中分散的窗口创建、事件泵、输入状态推进与渲染帧循环收束到 Core 层。
- 明确 Core 对 Platform、Render、Audio、Game 的统筹关系，为后续音频时钟同步、谱面推进、判定窗口和模块初始化顺序打底。

**设计原则**

- `Core::Application` 负责生命周期编排、模块初始化顺序、主循环驱动与退出流程，不直接承担 SDL/Vulkan 细节。
- `Platform` 继续负责窗口、事件泵、输入事件与输入状态；`Core` 消费项目自有事件和状态，不向上暴露 SDL 原始事件。
- `Render` 继续管理 Vulkan 资源与渲染帧提交；`Core` 只负责调用顺序、错误传播和退出协调。
- 时间系统优先提供 monotonic time、delta time、elapsed time 与 frame counter；先保持可变步长主循环，固定步长策略留到 Game/判定需求明确后再引入。
- 主循环先保持单线程、显式顺序执行：`beginFrame()`、拉取平台事件、更新输入状态、处理窗口事件、提交渲染、检查退出。
- 初始化和析构顺序必须可追踪，至少覆盖 SDL video 初始化、窗口创建、Renderer 初始化、主循环运行、Renderer idle、SDL shutdown。

**完成情况**

- [x] 新增 `include/Core/Application.hpp` 与 `src/Core/Application.cpp`，建立 `Raythm::Core::Application` 基础骨架。
- [x] 新增 `include/Core/Time.hpp` 与 `src/Core/Time.cpp`，封装 monotonic clock、delta time、elapsed time 与 frame counter。
- [x] 将 `src/main.cpp` 收缩为最小入口，仅负责创建 `Core::Application` 并调用 `run()`。
- [x] 在 `Application` 内接管 `Platform::Window`、`Platform::EventPump`、`Platform::InputState` 与 `Render::Renderer` 的创建、帧推进和退出流程。
- [x] 明确窗口关闭、`SDL_EVENT_QUIT`、focus lost、渲染失败、初始化失败等路径的退出语义与错误传播边界。
- [x] 为 Core 模块新增并注册 CTest，至少覆盖时间推进、frame counter、退出请求和最小生命周期流程。

**短期待办**

- [x] 先抽离 `src/main.cpp` 中现有最小循环到 `Core::Application::run()`，保持行为不变，优先完成职责收束。
- [x] 设计 `ApplicationOptions`、`ApplicationResult` 或同等轻量结构，集中表达窗口配置、退出结果和启动失败原因。
- [x] 为时间系统明确采样点：帧开始时间、上一帧 delta、累计运行时长和 frame index。
- [x] 固化 `InputState::beginFrame()`、事件消费、focus lost 清理、`Renderer::handleWindowEvent()` 与 `Renderer::renderFrame()` 的调用顺序。
- [x] 保持 Core 对 Platform/Render 的单向依赖，避免 Platform 或 Render 反向包含 Core。
- [x] 检查 Core 模块接入后 `CMakeLists.txt`、CTest 与 `.clangd` 是否需要同步更新。

**依赖关系**

- 依赖阶段 2 与阶段 4 已完成的 `Platform::Window`、`Platform::EventPump`、`Platform::InputState`。
- 依赖阶段 3 已完成的 `Render::Renderer` 最小清屏与 Present 骨架。
- 依赖当前 `src/main.cpp` 已验证的最小 SDL/Vulkan 主循环行为作为迁移基线。
- 后续 Audio、Game 模块将依赖本阶段输出的生命周期、时间推进和模块初始化边界。

**验证依据**

## 维护规则

- 每次开始开发前，先阅读 `docs/开发日志.md` 和本文档，确认当前已完成事项、遗留问题和短期计划。
- 完成计划项后，将对应任务标记为 `[x]`，并在 `docs/开发日志.md` 中记录完成内容、验证结果和后续计划。
- 完成模块实现、测试接入、架构调整、重要 Bug 修复或阶段性验证后，必须同步更新本文档的计划总览、对应阶段状态、完成情况、短期待办、验证依据和版本历史。
- 如果实际代码状态与本文档不一致，以代码和验证结果为准，并同步修正本文档与开发日志。
- 新增计划应优先落到短期计划或对应阶段计划中，避免只写在对话里而没有文档追踪。
