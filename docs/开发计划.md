### 阶段 4：Platform 输入事件与事件泵演进

**状态**：`[x]` 已完成窗口事件泵与设备输入封装，`[ ]` 输入状态系统待推进

**目标**

- 在不引入过重消息总线的前提下，建立键盘、鼠标与窗口生命周期事件的统一采集边界。
- 将 SDL 全局事件队列消费职责从 `Window` 中剥离，避免窗口层与输入层争用事件队列。
- 为后续 Core 主循环、输入低延迟处理、输入录制回放和动作映射保留稳定扩展点。

**设计原则**

- 优先新增薄 `Platform::EventPump` 或同等职责类型，而不是新增“总消息处理类”。
- `EventPump` 只负责从 SDL 拉取事件并翻译为项目自有事件，不承载业务决策。
- `Window` 继续作为 SDL 窗口 RAII 封装，长期职责应收敛到窗口资源、窗口 ID、尺寸/显示状态和 Vulkan surface 创建。
- 输入模块应维护 `InputState` 或同等状态快照，而不是只做按键消息回调。
- 当前阶段由 `main` 或未来 `Core::Application` 显式消费事件；仅当多个模块确实需要订阅同一事件流时，再考虑正式 dispatcher/router。

**短期待办**

- [x] 定义轻量窗口事件封装并通过 `Platform::EventPump` 覆盖 close、resize、focus 等窗口生命周期事件。
- [x] 定义轻量输入事件类型，覆盖 keyboard、mouse button、mouse motion。
- [x] 新增 `EventPump` 测试，验证 SDL 窗口事件翻译、未知事件跳过和退出事件处理路径。
- [x] 将 `Window::pollEvent()` 中的 SDL 取事件逻辑迁移到唯一事件泵，并同步更新 `src/main.cpp` 调用方。
- [ ] 新增输入状态系统测试，验证 key down/up、mouse button down/up、mouse motion 和帧边沿状态。
- [x] 在最小主循环中显式消费统一平台事件流；窗口事件已转发给渲染器，输入事件已进入 `PlatformEvent`，待后续接入输入状态系统。

**验证依据**

- `ctest --test-dir /e/RaythmDemo/build -C Debug --output-on-failure` 执行通过，3/3 tests passed。

## 维护规则

- 每次开始开发前，先阅读 `docs/开发日志.md` 和本文档，确认当前已完成事项、遗留问题和短期计划。
- 完成计划项后，将对应任务标记为 `[x]`，并在 `docs/开发日志.md` 中记录完成内容、验证结果和后续计划。
- 完成模块实现、测试接入、架构调整、重要 Bug 修复或阶段性验证后，必须同步更新本文档的计划总览、对应阶段状态、完成情况、短期待办、验证依据和版本历史。
- 如果实际代码状态与本文档不一致，以代码和验证结果为准，并同步修正本文档与开发日志。
- 新增计划应优先落到短期计划或对应阶段计划中，避免只写在对话里而没有文档追踪。
