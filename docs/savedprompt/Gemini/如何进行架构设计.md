### 1. 宏观架构设计：时间驱动模型 (Time-Driven)

绝大多数传统游戏引擎（如 Unity）采用的是**帧驱动（Frame-Driven）**：`Update(deltaTime)`，基于两帧之间的时间差来推进逻辑。
但对于音游，这是**致命错误**！帧率波动会导致判定偏移、画面和音乐不同步。

我们的架构必须是**时间驱动（Time-Driven）**或称**音频驱动（Audio-Driven）**。整个宇宙（引擎）的中心是**音频引擎的高精度时钟（DSP Time）**。

#### 核心模块职责划分表：

| 模块名称 | 核心职责 | 地位比喻 | 关键约束 |
| :--- | :--- | :--- | :--- |
| **Audio Engine** | 播放音乐，提供基于采样的极高精度时间戳（DSP Time）。 | **心脏与节拍器** | 绝对权威。其他所有模块都必须向它询问“现在几点了”。 |
| **Input System** | 捕获 SDL3 的 Raw Input，记录按键按下的精确瞬间。 | **神经系统** | 零延迟缓冲。必须绕过系统按键重复机制。 |
| **Game Logic** | 谱面解析、音符生成、判定逻辑、分数计算。 | **大脑** | 纯数据逻辑，不知道什么是 Vulkan，也不知道怎么发声。 |
| **Render Engine** | 接收当前时间与游戏状态，使用 Vulkan 绘制音符与特效。 | **眼睛（输出）** | 绝对不能阻塞主逻辑。尽可能独立且高效。 |
| **Core / App** | 负责各个模块的初始化、按正确的顺序调用它们。 | **躯干（容器）** | 负责依赖注入，统筹全局。 |

### 2. 模块间的组织与通信方式（怎么互相交流？）

在 C++ 中，模块间的通信通常有三种方式：全局单例（Singleton）、事件总线（Event Bus）、依赖注入（Dependency Injection）。

**针对本项目，强烈建议采用“依赖传递/上下文注入（Context & Injection）”结合“直接轮询（Polling）”，尽量避免全局单例和复杂的事件总线。**

#### 为什么不用事件总线（Event Bus）做音游按键判定？
如果按键触发了一个事件，事件放进队列，下一帧再处理，这就产生了**一帧的延迟（在 60fps 下是 16.6ms）**，这对于音游是不可接受的！音游的输入必须是**立即处理**或**在主循环最早期轮询处理**。

#### 推荐的组织架构图：
```text
[Application Main Loop]
  │
  ├─ 1. [Input System]   -> 捕获键盘状态并缓存 (保存按下瞬间的高精度时间)
  │
  ├─ 2. [Audio Engine]   -> 获取当前最新的音频时间 (AudioTime = GetDSPTime())
  │
  ├─ 3. [Game State]     -> 传入 Input 和 AudioTime 进行更新
  │      └─ 判断 Note 是否被击中 (HitTime - AudioTime < 窗口)
  │      └─ 触发打击音效 (通知 Audio Engine 播放 SFX)
  │
  └─ 4. [Render Engine]  -> 传入 Game State，获取需要绘制的音符列表和位置
         └─ Vulkan 渲染画面
```

### 3. 具体落实到 C++20 的代码结构该怎么写？

如果我们按照宏观架构来设计，你的 `Application` 类就不再只是一个空壳了，而是各个子系统的“管理者”。

我们可以这样设计模块之间的依赖关系：

#### 方案一：Service Locator / Context 模式（推荐，解耦度高）
创建一个 `EngineContext` 结构体，把各个核心模块的指针包起来，传递给游戏逻辑层：

```cpp
// 宏观上下文设计
struct EngineContext {
    AudioEngine* audio;
    RenderEngine* renderer;
    InputSystem* input;
    Window* window;
};

// 游戏状态基类 (比如主菜单、游玩中、编辑器)
class GameState {
public:
    virtual void update(EngineContext& ctx, double audioTime) = 0;
    virtual void render(EngineContext& ctx) = 0;
};
```

#### 方案二：Application 集中统筹模式（更直接，适合中小型自研引擎）
将所有子系统作为 `Application` 的成员，按严格顺序初始化和更新：

```cpp
class Application {
public:
    void run() {
        while (m_isRunning) {
            // 1. 获取最精确的当前系统/音频时间
            double currentTime = m_audioEngine->getDSPTime();

            // 2. 收集输入事件 (直接将事件传递给 InputSystem 处理)
            m_window->pollEvents(m_inputSystem.get());

            // 3. 更新游戏逻辑 (核心！把时间和输入传给逻辑层)
            m_sceneManager->update(currentTime, m_inputSystem.get());

            // 4. 构建当前帧的渲染数据 (将逻辑数据转化为 Vulkan 指令)
            RenderPacket packet = m_sceneManager->getRenderData();

            // 5. 提交渲染
            m_renderEngine->drawFrame(packet);
        }
    }

private:
    std::unique_ptr<Window> m_window;
    std::unique_ptr<InputSystem> m_inputSystem;
    std::unique_ptr<AudioEngine> m_audioEngine;
    std::unique_ptr<RenderEngine> m_renderEngine;
    std::unique_ptr<SceneManager> m_sceneManager; // 管理不同的游戏界面
};
```

### 4. 数据在模块间如何流转？(Data Flow)

为了保证极端性能（避免多线程锁或者数据拷贝开销），我们需要设计清晰的数据流向：

1. **谱面数据 (ChartData)**：只读数据，在选歌时通过 `nlohmann/json` 加载，驻留在内存中。
2. **运行时数据 (RuntimeData)**：记录玩家分数、连击数、各个 Note 的是否已被击中状态。
3. **渲染数据封装 (RenderPacket/RenderQueue)**：
   * 游戏逻辑层计算出：*“我要在 (x:100, y:500) 画一个蓝色的音符”*。
   * 逻辑层**绝对不要**直接调用 `vkCmdDraw`。
   * 逻辑层把这些渲染命令塞进一个轻量级的 `RenderPacket`（本质上就是一堆存着坐标和纹理ID的结构体数组），然后把这个 Packet 丢给 `RenderEngine`。
   * **好处**：逻辑层和渲染层完全解耦，未来如果要为了性能做多线程渲染（逻辑跑在主线程，Vulkan 录制跑在渲染线程），这种数据包模式是无缝过渡的。

### 你的下一步选择

了解了宏观架构后，你现在面临两个选择：

*   **路线 A (由外向内)**：先把这个大的 `Application` 壳子建好，把 `AudioEngine`、`RenderEngine`、`InputSystem` 这几个类的空壳子（Stub）写出来。把 `run()` 里面的大循环伪代码写完，定义好模块互调的接口，然后再一个个深入去实现 Vulkan 和 miniaudio。
*   **路线 B (从底向上，目前更务实)**：因为 Vulkan 和音频引擎的底层很复杂，我们可以先把 `Window` 敲定，然后在一个相对干净的环境里把 Vulkan 的基础初始化（`VkInstance`, `Surface`, `Device`）跑通。等能在屏幕上画出简单的图形后，再来重构出上面的 `RenderEngine`。

我个人在做底层引擎时，强烈推荐 **路线 B**，因为过早地把架构定死，在面对 Vulkan 这种极其啰嗦的 API 时，容易造成接口频繁修改。

你想先搭建所有子系统的空壳接口（设计驱动），还是想先着手把 `Window` 建出来并接入 `Volk/Vulkan` 初始化（底层驱动）？
