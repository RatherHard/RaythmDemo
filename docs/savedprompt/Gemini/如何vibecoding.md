“Vibe Coding”（意念编程 / 氛围编程）是随着大语言模型（尤其是 Claude 3.5 Sonnet 等具备超强代码理解能力的模型）和 Agent 工具（如 Claude Code、Cursor）普及而诞生的新概念。它的核心思想是：**人类提供架构蓝图、意图（Vibe）和审查，AI 负责敲代码、排错和实现细节。**

但是在**C++ 跨平台音游**这种极度吃性能、涉及多线程和底层硬件交互的项目中，如果只是随便跟 AI 说一句“帮我写个下落式判定”，AI 绝对会给你写出一堆内存泄漏、带锁阻塞、帧率暴跌的垃圾代码。

为了在你的项目中**高质量、无后遗症地运用 Vibe Coding**，你需要掌握以下一套“降维打击”的工作流：

---

### 第一步：构建“绝对真理”的上下文库（Context Engineering）

AI Agent 最怕的是“猜你的心思”。Claude Code 这种工具能够自动读取你本地的文件，因此你要做的第一件事是**把上文提到的技术文档喂给它**。

1. **建立 `.rules` 或 `system_prompt` 文件**
   在项目根目录新建一个 `PROJECT_RULES.md` 或 `.cursorrules`（如果你用 Cursor），强制要求 AI 在生成代码前必须遵守以下“军规”：
   > **示例：音游项目 AI 军规**
   > - 语言标准：严格遵循 C++20。
   > - 音频与判定：**绝对禁止**在音频线程和判定逻辑中使用 `std::mutex`，必须使用无锁编程（Lock-free queue）或原子操作（`std::atomic`）。
   > - 内存管理：禁止使用裸指针（裸 `new/delete`），必须使用智能指针或基于竞技场（Arena）的内存池，避免游戏循环中的内存碎片。
   > - 跨平台要求：优先使用 SDL3 处理平台差异，少写平台专有的 `#ifdef` 除非涉及底层硬件调用。
   > - 构建系统：所有的第三方库必须通过 CMake `FetchContent` 引入或放置在 `external/` 目录下静态链接。

2. **让 AI 阅读你的 TDD（技术设计文档）**
   当你启动 Claude Code 时，第一句话应该是：
   *“在开始写代码之前，请仔细阅读 `docs/01_Technical_Design_Document.md`，理解本项目基于 SDL3 + miniaudio + JSON 的跨平台音游架构。回复‘收到’确认理解。”*

---

### 第二步：任务极度拆解（Atomic Vibe Prompting）

不要让 AI “写一个游戏”，要让 AI “造一个零件”。将开发任务拆解为**原子级别**的模块，并给出清晰的输入输出（I/O）预期。

**❌ 错误的 Vibe 提示：**
> “帮我用 C++ 写一个读取 JSON 谱面的功能，然后生成 Note 让它们在屏幕上下落。”
> *(结果：AI 会把文件读取、解析、渲染逻辑全揉在一个巨大的面条代码文件里，后续根本没法改。)*

**✅ 高质量的 Vibe 提示（步骤分解法）：**
> **任务 1（先做）**：“使用 `nlohmann/json` 库，编写一个纯数据类 `ChartParser`。输入是一个 `.json` 文件的绝对路径，输出是一个包含所有 Note 的 `std::vector<NoteData>`。不用考虑渲染，只需要写出头文件和源文件，并提供一个使用断言（assert）的测试用例验证解析正确性。”
> 
> **任务 2（再做）**：“现在，基于刚才生成的 `ChartParser` 和给定的 DSP Time（`double current_dsp_time`），编写一个计算 Note 在屏幕上 Y 坐标的纯数学函数：`calculate_note_y(...)`，要求 BPM 变化时位置不能错乱。”

---

### 第三步：防御性 Review 与“反向质问”

C++ 的坑太多了。AI 生成完代码后，你作为架构师，必须进行“防御性审查”。

1. **强迫 AI 解释它的时间复杂度与线程安全性**
   当 Claude Code 生成了一段涉及底层时间的逻辑后，你可以这样质问它：
   *“你这段判定代码如果放在 1000Hz 轮询频率的独立输入线程中运行，会发生数据竞争（Data Race）吗？请证明你的安全性。”*
   *(AI 会立刻反思，如果它用了不安全的数据结构，它会自动道歉并重写为安全的无锁队列。)*

2. **利用编译器作为最终裁判**
   不要用眼睛去 Debug C++。让编译器和静态分析工具帮你：
   *在 CMake 中开启 `-Wall -Wextra -Werror`（所有警告视为错误）。*
   如果编译失败，直接把控制台的报错信息整坨复制给 Claude Code：
   *“CMake 报了如上错误，好像是 SDL3 的链接问题，请修复 `CMakeLists.txt`。”*

---

### 第四步：巧妙利用 Agent 的“重构”能力（Refactoring）

在 Vibe Coding 中，你不用一开始就追求代码的完美结构。你可以先让 AI 写出“能跑的脏代码”，然后再通过 Vibe 驱动它重构。

**实战场景：将硬编码抽离**
1. 最初你可以让 AI 快速糊一个窗口和画矩形的逻辑，全部写在 `main.cpp` 里测试渲染。
2. 看到屏幕上成功画出东西后，下达重构指令：
   *“现在的 `main.cpp` 太臃肿了。请你将它重构为面向对象的结构。提取出一个 `Application` 类负责生命周期，一个 `Renderer` 类封装所有的 SDL_Render 命令。保留所有现有功能不变，更新 CMake 配置。”*
3. Claude Code 这种工具会自动帮你创建新文件、迁移代码并修改头文件包含。

---

### 五、 针对音游核心难点的专属 Prompt 技巧

在你的项目中，有三个地方极其容易翻车，建议使用以下特定的 Prompt “护体”：

1. **针对【音画同步】模块：**
   > “现在要实现音画同步的核心时间轴。**注意！绝对禁止使用 `Time.deltaTime` 累加的方法。** 唯一的时间基准必须通过请求 `miniaudio` 的内部帧数或 DSP 时钟来获取，将音乐当前播放的时间戳作为全局唯一真实时间（Source of Truth），画面的渲染进度必须完全拟合这个时间戳。”

2. **针对【打击判定】模块：**
   > “编写 `InputSystem`。我需要最低延迟的判定。当操作系统触发触屏/按键中断的瞬间，不要等下一帧渲染！立刻捕获当前硬件时间并调用判定回调。设计时考虑将判定逻辑与渲染逻辑彻底解耦。”

3. **针对【谱面编辑器界面 (ImGui)】：**
   > “使用 Dear ImGui 编写编辑器的时间轴面板。要求时间轴可以无限缩放（类似宿主软件 FL Studio），由于谱面可能有几千个 Note，要求实现**视椎体剔除（Culling）**逻辑，只循环和绘制当前屏幕可视时间范围内的 Note 和音频波形，否则会导致帧率下降。”

### 总结：Vibe Coding 的心态转变

在过去，你是**“写代码的工人”**，编译器是你的老板。
在 Vibe Coding 的时代，你是**“架构师和包工头”**，AI（Claude Code）是你的工人，C++ 编译器是你的质检员。

不要把时间花在手敲 `std::vector::iterator` 这种繁文缛节上，要把精力百分百集中在**架构设计、组件解耦、多线程时序和游戏判定手感调优**上。这才是最高质量的 AI 辅助开发之道。
