### 修复运行时卡死与音频/谱面不同步

**完成内容**

- 扩展 `tests/core_gameplay_runtime_test.cpp`，新增基于 frame song time、frame SDL timestamp 与 input event timestamp 的 `LanePressEvent` 映射回归测试，覆盖延迟处理输入应回映到更早 song time、future timestamp clamp、过旧 timestamp clamp 和 repeat key 忽略。
- 扩展 `Platform::InputEvent`，在 `include/Platform/EventPump.hpp` 中保留输入事件 SDL nanosecond timestamp，并在 `src/Platform/EventPump.cpp` 的键盘、鼠标按键和鼠标移动翻译路径中传递该时间戳。
- 更新 `Core::GameplayRuntimeDriver`，新增 timestamp-aware `makeLanePressEvent()` overload，由 `frameSongTime - (frameTimestamp - eventTimestamp)` 计算事件发生时的 song time，并 clamp 到 `[0, frameSongTime]`，避免使用“事件被处理时”的音频时间导致判定偏晚。
- 更新 `src/Core/Application.cpp`，主循环每帧只采样一次 `AudioEngine::getPlaybackTimeSeconds()` 和 `SDL_GetTicksNS()` 作为输入映射基准，事件处理阶段不再为每个输入事件即时采样音频时间。
- 扩展 `tests/render_strategy_test.cpp`，新增 fence wait、swapchain acquire、present result 的纯策略分类测试，覆盖 timeout、out-of-date、suboptimal、device lost 与 fatal error。
- 在 `RenderStrategy` 中新增 `RendererWaitStatus` 与 `RendererFrameStatus`，并将 `Renderer::renderFrame()` 从 bool 结果改为可观测状态；把 `vkWaitForFences()` 与 `vkAcquireNextImageKHR()` 的 `UINT64_MAX` 无限等待改为有限 timeout，`VK_TIMEOUT` 返回 `RenderStalled`，device lost 返回 fatal 状态。
- 调整 swapchain recreate 与 shutdown 路径，不再在正常运行路径直接调用无限 `vkDeviceWaitIdle()`；`Core::Application` 对 `NoDrawable`、`RecoveringSwapchain`、`RenderStalled` 保持短暂 delay 后继续泵 SDL 事件，对 `DeviceLost` / `FatalError` 进入失败路径。

**验证结果**

- 修复前新增回归测试可稳定进入红色状态：`RaythmDemoCoreGameplayRuntimeTest` 因 `InputEvent` 缺少 timestamp 与 `GameplayRuntimeDriver::makeLanePressEvent()` 缺少 frame timestamp overload 编译失败。

**后续计划/遗留问题**

### 建立最小 Core 运行时游玩闭环

**完成内容**

- 新增 `include/Core/GameplayRuntimeDriver.hpp` 与 `src/Core/GameplayRuntimeDriver.cpp`，建立 Core 侧运行时驱动 seam，将 `Platform::InputState`、`Game::LaneInputMapper`、`Game::GameplaySession`、`Game::GameplayViewAdapter` 串接为每帧 `Render::RenderCommand` 输出。
- `GameplayRuntimeDriver` 支持按当前 viewport 重建玩法视图布局，使轨道、判定线和 note 命令随窗口/可绘制尺寸变化，而不把 Game 逻辑绑定到 Vulkan 或 SDL 细节。
- 扩展 `Core::ApplicationOptions`，新增 path-typed `assetRoot` 与 `startupChartPath`，默认使用 `assets/charts/00001/00001.json` 作为启动谱面，并通过 CMake `POST_BUILD` 将 `assets/` 复制到 `RaythmDemo` 可执行文件目录，避免运行时依赖进程 CWD。
- 在 `Core::Application::run()` 中接入启动谱面加载、音频加载、`AudioEngine` 播放启动、每帧读取 audio playback time、采样输入、更新 gameplay runtime、提交 2D render commands 和渲染帧。
- 为运行时音频加载新增 `AudioEngine::loadMusicFromMemory(std::vector<std::byte>&&)`，Core 先按 chart loader 同类策略执行 no-follow/open-then-verify 读取、assetRoot confinement、Ogg signature 校验和大小限制，再将已验证音频字节移动给 miniaudio memory decoder，避免校验后按路径重新打开导致 TOCTOU 风险。
- 主循环在 drawable size 为 0 或渲染暂停时暂停音频并跳过 gameplay 推进，恢复可绘制尺寸后再恢复音频，避免最小化期间谱面继续推进并批量 Miss。
- 在事件处理阶段把绑定键的 key press 转换为 timestamped `LanePressEvent`，并由 `GameplayRuntimeDriver` 调用 `GameplaySession` 的 timestamped update 路径，降低帧末采样导致的同帧密集输入丢失风险；无 timestamped event 时仍保留帧级 `LaneInputSnapshot` 路径。
- 新增 `tests/core_gameplay_runtime_test.cpp` 与 `RaythmDemoCoreGameplayRuntimeTest`，覆盖 baseline/note commands、默认 D/F/J/K 输入判定、held lane 视觉、viewport resize、timestamped dense same-lane press events、bound key event 转换、非法 viewport 和自动 Miss sweep。
- 更新 `CMakeLists.txt`，将 `GameplayRuntimeDriver` 接入 `RaythmDemoCore`，使 Core 链接 Audio/Game/Render，并注册新的 CTest 目标。

**验证结果**

- `cmake --build build --config Debug && ctest --test-dir build -C Debug --output-on-failure` 执行通过，11/11 tests passed。
- 构建输出确认 `RaythmDemo` POST_BUILD 步骤执行 `Copy runtime assets next to RaythmDemo executable`，运行时资源已复制到 `build/Debug/assets`。
- 已使用 `code-reviewer`、`cpp-reviewer` 与 `security-reviewer` 多轮审查；已修复启动资产路径依赖 CWD、音频路径 TOCTOU、音频 decoder 输入范围、timestamped 输入丢失、最小化期间 gameplay 继续推进、chrono 边界、内存音频双缓冲和 Windows/Linux 文件读取健壮性等问题，最终复审无阻塞问题且安全复审无发现。

**后续计划/遗留问题**
